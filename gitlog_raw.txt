    Fix misnomer: rename HANDOVERPKT to BEACONPKT
    
    It was a clear misnomer, as it does NOT announce or request handover.
    
    It is a broadcast message that eNodeBs periodocally send, and UEs
    utilize it measure signal quality to different eNodeBs, and use it
    to decide when to handover to a different eNodeB.
    
    It has no direct equivalent in actual LTE/5G, but represents/combines
    multiple signals / protocol elements into a simulation abstraction.

    as before. No fingerprint change.
    
    Subsequent commits move the DATA into HandoverController too, and clean
    up the mess.

      collects RX entity output and forwards to upper layer.

      collects TX entity output and routes to correct RLC gate

      mode switch notifications. Owns rxEntities_ and bypassTxEntities_
      maps and their creation/deletion.

      API (create/delete/lookup/activeUeUL) to UpperMux and LowerMux.
      External callers (Rrc, HandoverController, LteMacEnb) continue
      to reference PdcpMux via ModuleRefByPar<PdcpMux> unchanged.
    
    Cross-module gate wiring: UpperMux wires TX entity output gates to
    LowerMux.fromTxEntity[]; LowerMux wires RX entity output gates to
    UpperMux.fromRxEntity[]. Bypass entities are fully owned by LowerMux.
    
    LtePdcp.ned stripped to facade (no gates, only isNR/hasD2DSupport
    params retained for external module compatibility).
    
    PdcpLayer.ned updated with upperMux + lowerMux + mux submodules.

    LteSchedulerEnb: Fix multi-UE starvation in multi-DRB scheduling
    
    In scheduleGrant(), when a node already had a codeword allocated,
    the "BAD patch" set terminate=true, which aborted the entire
    scheduling round. This prevented any other node's CIDs from being
    processed, causing UEs beyond the first to receive zero DL traffic
    in multi-DRB scenarios (e.g. with QOS_PF on the sdap branch).

    for remaining CIDs belonging to other nodes.

    not triggered in Unipisa/Simu5G#294.

Author: Andras Varga <andras@omnetpp.org>

    QoSAwareScheduler: replace greedy grant loop with proportional fairness
    
    Previously, prepareSchedule() granted UINT32_MAX bytes to the top-scored
    CID, letting the highest-priority node consume all available RBs before
    other nodes got any. This is strict priority, not proportional fair.
    
    New design splits prepareSchedule() into three phases:
    
    Phase 1 - Score: compute QoS-weighted PF score and available bytes for
    each eligible CID (unchanged scoring formula).
    
    Phase 2 - Quota: select the best CID per node (since scheduleGrant
    allows only one grant per node per TTI due to the codeword allocation
    constraint). Compute each node's proportional share of bandwidth:
      quota_i = availableBytes_i * (score_i / totalScore)
    where totalScore is the sum of best-per-node scores only.
    
    Phase 3 - Grant: process CIDs in score order, but cap each grant to
    the node's proportional quota instead of UINT32_MAX. Non-best CIDs
    of already-granted nodes get eligible=false from scheduleGrant
    automatically.
    
    This ensures that bandwidth is distributed proportionally to the
    QoS-weighted PF metric across nodes within each TTI, rather than
    being monopolized by the top-scored node.
    
    Testing proportional fairness under contention:
    
    Used the VoIP-DL-MultiQfi-Heavy config (1000B packets at 1ms interval,
    no silences, synchronized start) to create sustained contention between
    two nodes (2049, 2050) with identical QoS configurations.
    
    With the old greedy loop (requestGrant(cid, UINT32_MAX)), the top-scored
    node would consume all 7250B of available bandwidth each TTI, starving
    the other node until the next TTI.
    
    With the new proportional quota mechanism:
    
    $ cd simulations/nr/standalone
    $ simu5g_dbg -u Cmdenv -c VoIP-DL-MultiQfi-Heavy omnetpp_drb.ini \
        --sim-time-limit=0.1s --cmdenv-express-mode=false 2>&1 \
        | grep -E "Quota|Grant cid" | head -20
    
    0.003 QoSAwareScheduler::Quota node=2049 cid=MacCid(nodeId=2049, lcid=1) score=3e+07 share=0.5 quota=3625B
    0.003 QoSAwareScheduler::Quota node=2050 cid=MacCid(nodeId=2050, lcid=3) score=3e+07 share=0.5 quota=3625B
    0.003 QoSAwareScheduler::Grant cid=MacCid(nodeId=2049, lcid=1) limit=3625 granted=3625B
    0.003 QoSAwareScheduler::Grant cid=MacCid(nodeId=2050, lcid=3) limit=3625 granted=3625B
    0.003 QoSAwareScheduler::Grant cid=MacCid(nodeId=2050, lcid=2) limit=4294967295 granted=0B
    0.004 QoSAwareScheduler::Quota node=2049 cid=MacCid(nodeId=2049, lcid=0) score=1e+07 share=0.5 quota=3625B
    0.004 QoSAwareScheduler::Quota node=2050 cid=MacCid(nodeId=2050, lcid=2) score=1e+07 share=0.5 quota=3625B
    0.004 QoSAwareScheduler::Grant cid=MacCid(nodeId=2049, lcid=0) limit=3625 granted=3625B
    0.004 QoSAwareScheduler::Grant cid=MacCid(nodeId=2050, lcid=2) limit=3625 granted=3625B
    0.004 QoSAwareScheduler::Grant cid=MacCid(nodeId=2049, lcid=1) limit=4294967295 granted=0B
    
    So:
    
    - Both nodes get share=0.5, quota=3625B each (50/50 split of 7250B)
    - Each node's best CID is granted exactly 3625B per TTI
    - The second CID of each node gets eligible=false (one grant per node
      per TTI, enforced by scheduleGrant)
    - The PF EWMA naturally alternates which CID per node is "best" each TTI
     (the just-served CID's pfRate_ increases, lowering its score next time)
    
    All four DRB configs (VoIP-DL, VoIP-DL-MultiQfi, VoIP-DL-MultiQfi-Heavy,
    VoIP-UL) run to completion without errors.
